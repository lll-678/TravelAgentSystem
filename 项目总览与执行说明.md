# 项目总览与执行说明

说明：这是 TravelAgentSystem 当前的统一交接与执行文档，后续 agent 优先读取这一份。

## 1. 项目定位

构建一个前后端分离的个性化旅游系统，覆盖“查询-规划-推荐-记录-展示”全流程场景，支持从地点搜索到路线生成再到内容沉淀的闭环。

## 6. 需要持续遵守的约束

### 6.1 技术红线
1. 核心算法手写：Graph/Dijkstra、Trie、Heap、Huffman、倒排索引。
2. 查询主路径不得退化为 O(n) 全表扫描。
3. 距离排序以图路径距离为准，不以直线距离直接替代。
4. 接口字段变更必须同步更新接口文档。

### 6.2 目录职责
1. `app/api`：接口入口与协议层。
2. `app/services`：业务流程编排与策略组合。
3. `app/core`：手写算法与数据结构。
4. `app/db`：数据库模型、连接与数据访问。
5. `app/models`：Pydantic 请求/响应模型。

## 7. 协作与开发节奏

### 7.1 三人分工
1. 算法负责人：Graph / Trie / Heap / TSP / Huffman。
2. 后端负责人：FastAPI 路由、Service、DB、CRUD。
3. 前端与联调负责人：Vue 页面、地图可视化、接口接入。

### 7.2 协作规则
1. 使用 feature 分支开发。
2. 每个任务先有 Issue，再发 PR。
3. PR 至少一个人 Review 后再合并。
4. 主分支只保留可发布代码。

## 8. 三周里程碑

1. 第 1 周：骨架与数据结构，完成 Graph / Trie / Heap 初版。
2. 第 2 周：核心功能联调，完成路径规划与推荐接口闭环。
3. 第 3 周：扩展与交付，补倒排索引、Huffman、测试和答辩材料。

## 9. TripStar 对照计划

目标不是复刻 TripStar，而是借它的分层方式来整理本项目。

1. 保留：清晰的壳层、稳定的页面结构、统一样式体系。
2. 简化：不必要的重复组件、过度复杂的页面碎片化。
3. 自研：路径规划、推荐、日记检索、压缩等核心能力。

后续如果继续做架构对照，建议优先输出：
1. TripStar vs 本项目的模块对照表。
2. 固定版 MVP 接口清单。
3. 下一轮任务看板（到人、到文件、到接口）。

## 10. 2026-05-09 起的统一执行方案

本章从 2026-05-09 起生效，优先级高于前文中分散的历史计划描述。后续 agent 进入仓库后，应先按本章判断目标、边界和开发顺序。

### 10.1 当前统一目标

本项目的目标正式收敛为：

**TripStar-lite 的 AI 规划系统 + 课程要求的手写算法实现。**

这意味着项目不再追求一次性覆盖“旅游平台全部能力”，而是优先做出一条真实、可演示、可答辩、可持续迭代的最小闭环。

### 10.2 第一优先级：先打通一条真实闭环

当前最重要的不是继续扩展功能面，而是打通下面这条用户闭环：

1. 用户在首页输入城市、日期、偏好。
2. 后端根据真实数据生成旅行计划。
3. 结果页展示每日行程、景点卡片、预算摘要。
4. 地图展示景点与路线。
5. AI Chat 能基于当前行程回答追问。

只要这 5 步稳定跑通，本项目就已经具备 `TripStar-lite` 的最小产品形态。

### 10.3 课程要求与算法落位

课程要求不是独立于产品之外的“附加作业”，而是要嵌入主闭环中：

1. `Trie`：用于景点搜索、前缀匹配、候选补全。
2. `Heap`：用于候选景点打分后的 Top-K 筛选。
3. `Graph + Dijkstra`：用于景点图建模、距离计算与路线规划。
4. `Huffman`：不进入第一阶段主闭环，作为第二阶段扩展项。
5. 倒排索引：不进入第一阶段主闭环，作为第二阶段扩展项。

### 10.4 当前版本必须收缩的范围

为避免继续目标发散，当前版本明确做以下收缩：

1. 暂不把“日记系统”作为第一阶段主线。
2. 暂不把“知识图谱可视化”作为第一阶段主线。
3. 暂不把“小红书内容接入”升级为独立大模块推进；它只作为 P1 的内容增强输入。
4. 暂不把“复杂多智能体编排”作为第一阶段主线。
5. 暂不把“全面推荐系统”作为独立大模块推进。
6. 暂不继续扩展大量未落地的假接口和占位接口。

这些内容可以保留为后续扩展方向，但不能阻塞第一条真实闭环。

### 10.5 MVP 边界（第一阶段可交付版本）

第一阶段 MVP 只包含以下能力：

1. 首页表单：城市、日期、偏好、出行方式、住宿偏好。
2. 行程生成接口：根据输入生成可展示的旅行计划。
3. 景点候选筛选：使用手写算法完成候选检索和 Top-K。
4. 路线展示：在地图中展示景点与路径。
5. 行程结果页：展示每日安排、预算、景点简介。
6. 行程问答：对当前计划进行 AI 问答。

第一阶段 MVP 不要求：

1. 多智能体并发工作流。
2. 小红书内容接入（非深度抓取）。
3. 知识图谱正式版。
4. 日记 CRUD、倒排索引、压缩系统正式接入。
5. 大规模生产级推荐引擎。

### 10.6 P0 改造顺序

后续开发必须优先按下面顺序推进，不得跳步做边缘功能。

#### P0-1 文档与接口收敛

目标：消除“文档描述”和“实际代码”不一致的问题。

要求：
1. 把不存在的接口、模型、服务从主文档中移除或标记为后续扩展。
2. 明确当前第一阶段只保留：`/api/settings`、`/api/pois`、`/api/routes`、`/api/trips`、`/api/chat/ask`。
3. 所有前端调用的接口都必须在后端存在；所有后端公开接口都必须在文档中可查。

#### P0-2 行程生成闭环

目标：让 `/api/trips` 从“演示数据生成器”升级为“真实闭环入口”。

要求：
1. 接收城市、日期、偏好等输入。
2. 从本地数据源中筛选候选 POI。
3. 用 `Heap` 做候选打分后的 Top-K 选择。
4. 用天数分配景点，输出结构化日程。
5. 为结果页提供预算、建议和必要的景点元数据。
6. 输出结构必须稳定，作为前端唯一可信数据源。

#### P0-3 结果页去演示化

目标：让 `Result.vue` 从演示骨架变成真实结果页。

要求：
1. 去掉硬编码日期和演示文案。
2. 去掉无计划时自动拼装 demo 数据的逻辑。
3. 页面只渲染真实接口返回的数据。
4. 地图、日程、景点概览都以同一份 `trip plan` 为数据源。

#### P0-4 AI Chat 最小接入

目标：做出 `TripStar-lite` 所需的最小 AI 问答闭环。

要求：
1. 前端把当前 `trip_plan`、用户问题和历史消息发送给后端。
2. 后端返回基于行程上下文的回答。
3. 第一阶段不强求多智能体，只要求“能围绕当前计划回答问题”。

### 10.7 P1 扩展顺序

在 P0 稳定之后，再考虑下面内容：

1. 搜索增强：更完整的 `Trie` 搜索体验和候选提示。
2. 推荐增强：引入更明确的偏好打分模型。
3. 小红书内容接入：作为外部内容源增强 POI、推荐和行程理由。
4. 知识图谱：从结果数据构建轻量图谱展示。
5. 日记、倒排索引、Huffman：作为课程扩展项逐步接入。
6. 多智能体：当单 Agent 闭环稳定后再拆分角色。

### 10.8 Agent 执行规则

后续 agent 进入仓库后，应遵守以下规则：

1. 优先服务 P0 主闭环，不得优先实现边缘展示功能。
2. 如果发现文档、接口、前端、后端不一致，应先收敛再扩展。
3. 如果一个能力只有占位 UI、没有真实数据流，不得视为“已完成”。
4. 如果一个接口只返回 demo 数据，不能标记为“闭环已完成”。
5. 如果课程算法与产品功能冲突，应优先把算法嵌入真实主流程，而不是另起一套孤立实现。

### 10.9 当前建议的最近执行任务

按优先级排序，下一轮应直接做：

1. 完成 P1 计划板块草案并确认优先级。
2. 继续打磨结果页响应式和地图展示细节。
3. 补充更完整的联调/验收脚本，覆盖主闭环。
4. 若后续再扩展能力，优先按 P1 计划推进。

### 10.10 P1 计划板块（草案）

P1 的目标不是继续扩张页面数量，而是在 P0 闭环稳定后，把“体验完整度”和“课程扩展能力”补齐。

#### P1-1 搜索增强

目标：把 POI 搜索从基础查询升级为更顺滑的候选补全体验。

要求：
1. 前缀补全更准确，支持更自然的中文输入。
2. 搜索结果按相关性和城市匹配度排序。
3. 首页或搜索入口能直接帮助用户定位可选景点。

#### P1-2 小红书内容接入

目标：把小红书公开内容作为外部信息源，增强 POI 和推荐质量。

要求：
1. 只接入公开内容，不做高频深度抓取。
2. 提取标题、地点、标签、图片、核心亮点和避坑信息。
3. 将内容回流到景点卡片、推荐理由和行程摘要。
4. 失败时可自动降级到本地样例数据，不阻塞主闭环。

任务拆分：
1. 先定义内容数据结构，统一笔记、地点、标签、图片和摘要字段。
2. 再补服务层入口，把小红书内容接入到 POI / 推荐 / 行程理由生成链路。
3. 然后在前端结果页展示“内容来源”和“推荐理由”，不改变主闭环结构。
4. 最后补最小测试，覆盖内容解析、降级逻辑和前端字段显示。

对应 Issue 清单见：`docs/ISSUE_BACKLOG_P1.md`

当前实现口径：
1. 小红书能力定位为“内容增强层”，不阻塞 `POST /api/trips` 主闭环。
2. 默认使用内置本地样例；若配置 `XHS_SAMPLE_NOTES_PATH`，则优先读取外部样例内容。
3. 读取失败或无匹配内容时自动降级到本地样例，并在结果页展示来源与推荐理由。
4. 当前还支持通过前端运行时设置直接导入外部 JSON 样例，形成“外部文件输入 -> 服务端保存 -> `/api/trips` 命中 -> 结果页展示”的完整闭环。
5. 外部 JSON 不再局限于规范化 notes 数组，当前已兼容 `TripStar` 风格搜索/详情 bundle 与常见第三方中间结果结构。

#### P1-3 推荐增强

目标：让推荐逻辑更贴近用户偏好，而不是只做静态筛选。

要求：
1. 结合偏好、城市、出行方式做更细的打分。
2. 给出可解释的推荐理由。
3. 保持结果页可展示、可追问、可复用。

#### P1-4 轻量知识图谱

目标：把结果页数据进一步组织成关系网络，而不是独立大模块。

要求：
1. 从当前 trip plan 生成轻量关联结构。
2. 只做展示层，不抢占主闭环优先级。
3. 不引入复杂编排和额外维护成本。

#### P1-5 课程扩展项

目标：把课程要求的扩展算法逐步接回主流程。

要求：
1. 日记、倒排索引、Huffman 作为扩展能力逐步接入。
2. 每个能力都要有明确入口和可验证结果。
3. 不允许形成脱离产品主线的孤立 demo。
