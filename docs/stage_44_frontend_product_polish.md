# Stage 44 Frontend Product Polish

## Goal

提升前端成熟度，让系统更像可演示的产品级旅游算法平台，而不是课程作业页面集合。本阶段只优化视觉层、信息架构和页面交互呈现，不修改后端接口契约和核心算法逻辑。

## TripStar Reference Takeaways

- 入口要少而清晰：主页面应优先呈现数据驾驶舱、核心服务入口和演示链路。
- 页面要有稳定工作区：左侧控制/筛选，右侧地图、结果或详情，避免控件散落。
- 结果要卡片化和摘要化：表格保留给管理场景，面向用户的结果优先展示名称、状态、评分、距离、热度和下一步动作。
- 状态要可解释：算法 trace、数据来源、当前用户、地图图层和结果规模要有清晰位置。
- 视觉要克制：使用产品后台风格的色彩、分隔、阴影和密度，不做纯营销式落地页。

## Deliverables

- 全局设计语言：统一背景、侧边栏、顶部栏、卡片、表格、按钮、输入控件和 trace 面板。
- 应用外壳：侧边导航增加模块说明和状态感，顶部栏显示当前模块、运行环境、账号状态。
- 首页/答辩演示页：强化驾驶舱、指标、算法能力和演示链路的产品呈现。
- 地图/路线/附近设施：形成工具台布局，明确控制区、地图区、结果摘要和算法/路线状态。
- 地图组件：增加图层状态条和更专业的 fallback 面板。
- Harness 文档：更新 feature matrix、acceptance checklist 和 README 当前阶段说明。

## Non-goals

- 不新增业务功能。
- 不重写接口或改变请求参数。
- 不替换 Element Plus，也不引入大型 UI 依赖。
- 不把 TripStar 的代码或视觉风格直接复制到本项目。

## Acceptance

- 主要页面视觉风格统一，页面之间没有明显的课程项目式割裂。
- 地图、路线、附近设施页面能清楚区分控制台、地图画布和结果信息。
- 首页和答辩演示页能直接说明数据规模、算法能力和演示链路。
- `npm run typecheck`、`bash scripts/check_frontend.sh`、`bash scripts/check_all.sh` 通过。
